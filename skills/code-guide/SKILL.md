# Karpathy Guidelines

> "模型会代你做错误的假设，然后不假思索地执行。它们不管理自身的困惑，不寻求阐明，不提出矛盾，不展示权衡，在应该提出异议时也不反驳。"
>
> "他们真的很喜欢把代码和 API 搞复杂，堆砌抽象概念，不清理死代码……明明 100 行能搞定的事情，非要实现成 1000 行的分析架构。"
>
> "他们有时仍然会列出或删除自己理解不足的代码和注释，即使这些内容与任务本身相关。"

## 解决方案

四个原则，集中在一个文件中，直接解决这些问题：

| 原则 | 解决什么问题 |
|---|---|
| 编码前思考 | 错误想象、隐藏困惑、物资权衡 |
| 简单优先 | 过度复杂、归纳总结 |
| 精准修改 | 无关编辑、接触不宜接触的代码 |
| 目标驱动执行 | 通过测试优先、可验证的成功标准 |

## 四个原则详解

### 1. 编码前思考

**不要设想。不要隐藏。用于平衡。**

LLM 经常默默地选择一种解释然后执行。这个原则明确强制推理：

- 明确说明假设——如果不确定，询问而不是猜测
- 提出多种解释——当出现歧义时，不要默默选择
- 适时提出异议—如果存在更简单的方法，出来说
- 暂停时间停止来——指出节日的地点并要求澄清

### 2. 简洁优先

**使用最少的代码解决问题。不要过度推测。**

对抗过度工程的倾向：

- 不要添加要求额外的功能
- 不要作为总体代码创建抽象
- 不要添加未要求的"灵活性"或"可配置性"
- 不要为不可能发生的场景做错误处理
- 如果 200 行代码可以写成 50 行，重写它

检验标准：资深工程师会觉得这过于复杂吗？如果是，简化。

### 3. 精准修改

**只碰必须碰的。只清理自己造成的破坏。**

编辑现有代码时：

- 不要"改进"相关的代码、注释或格式
- 不要改造没坏的东西
- 符合现有风格，即使你更倾向于不同的写作法
- 如果注意到无关的死代码，提一下——不要删除它

当你的父母产生孤儿代码时：

- 删除因你的事件而整合无用的导入/变量/函数
- 不要删除预先存在的死代码，除非被要求

检验标准：每一行都应该能够直接间接用户的修改请求。

### 4. 目标驱动执行

**定义成功标准。循环验证直至达成。**

将指令式任务转化为可验证的目标：

| 不要这样做... | 转化为... |
|---|---|
| "添加验证" | "为无效输入编写测试，然后让它们通过" |
| "修复错误" | "编写重写 bug 的测试，然后修复" |
| "重构 X" | "确保重构前后测试贯穿" |

对于步骤多任务，说明一个简短的计划：

```
1. [步骤] → 验证: [检查]
2. [步骤] → 验证: [检查]
3. [步骤] → 验证: [检查]
```

无力的成功标准让 LLM 能够独立循环执行。弱标准（"弥补工作"）需要不断阐明。
