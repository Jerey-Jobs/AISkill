## 代码解释

当用户请求解释代码时，请遵循以下原则：

### 解释结构
1. 概述：简要说明代码的整体功能和目的
2. 核心逻辑：逐步解析关键代码段的工作原理
3. 数据流：描述数据如何在代码中流动和转换
4. 依赖关系：说明与其他模块/函数的交互

### 解释风格
- 使用中文进行解释
- 避免逐行翻译，聚焦于逻辑和意图
- 对复杂算法使用类比或图示说明
- 标注潜在的边界情况和异常处理

### Rust 特定说明
- 解释所有权和借用的使用场景
- 说明生命周期标注的必要性
- 解析 trait 实现和泛型约束
- 阐述 async/await 和并发模式

### Flutter/Dart 特定说明
- 解释 Widget 树结构和状态管理
- 说明 BuildContext 的作用域
- 解析异步操作和 Future 处理

## 代码优化

当用户请求优化代码时，请遵循以下原则：

### 优化分析流程
1. 识别问题：定位性能瓶颈或代码异味
2. 评估影响：分析优化的收益和风险
3. 提出方案：给出具体的优化建议
4. 验证结果：说明如何验证优化效果

### 性能优化方向

#### Rust 性能优化
- 减少不必要的内存分配（避免频繁 clone）
- 使用迭代器链代替显式循环
- 合理使用 `&str` vs `String`
- 利用 `Cow<str>` 延迟克隆
- 使用 `parking_lot` 替代标准库锁
- 批量操作代替单次操作
- 使用 `rayon` 进行并行处理
- 避免不必要的 `Box<dyn Trait>`，考虑枚举

#### 异步优化
- 避免在异步代码中阻塞
- 合理使用 `tokio::spawn` 和 `spawn_blocking`
- 使用 `join!` 并发执行独立任务
- 避免持有锁跨越 await 点

#### 数据库优化
- 使用批量插入/更新
- 合理使用事务
- 优化 SQL 查询，避免 N+1 问题
- 使用索引覆盖查询

#### Flutter/Dart 优化
- 减少 Widget 重建范围
- 使用 `const` 构造函数
- 合理使用 `ValueListenableBuilder`
- 避免在 build 方法中进行耗时操作
- 避免改动Widget引起动画异常
- 性能至上，避免一切不必要的CPU计算
- 一切的优化都要关注性能，并注释好为什么这么改
- 评估问题的时候，可以去查看flutter源码，确认下是否有这种问题，不要瞎猜
- 改动记得检查编译错误，别有什么找不到import,参数不对的情况

### 代码质量优化

#### 可读性
- 提取复杂条件为命名变量或函数
- 使用有意义的变量和函数名
- 保持函数职责单一
- 添加必要的注释说明意图

#### 可维护性
- 遵循 DRY 原则，提取重复代码
- 使用设计模式解决常见问题
- 保持模块边界清晰
- 编写自解释的代码

#### 错误处理
- 使用 `Result` 和 `Option` 而非 panic
- 提供有意义的错误信息
- 在适当层级处理错误
- 使用 `thiserror` 定义错误类型

### 输出格式
- 使用中文描述优化建议
- 提供优化前后的代码对比
- 说明优化的原因和预期效果
- 标注可能的副作用或注意事项
